# ObjectScript Quest Master — Project Specification

> **Purpose**: This document is a complete specification for an AI-powered gamified learning app for InterSystems ObjectScript. It is designed to be fed directly to Claude Code as a project brief.

---

## 1. Overview

**Quest Master** is a browser-based app that teaches InterSystems ObjectScript through a gamified quest system. The user writes real ObjectScript code in a built-in editor. Code is compiled and executed on a live IRIS instance (running in Docker). Quest generation and code review are powered by the Claude API.

### Core Loop

1. Player receives a quest (generated by Claude API)
2. Player writes ObjectScript in an in-browser code editor
3. Code is sent to IRIS via REST → compiled → executed
4. Output is sent to Claude API for evaluation
5. Player receives XP, feedback, and unlocks the next quest

### Key Principles

- **Real execution**: All code runs on a real IRIS instance. No simulations.
- **Progressive difficulty**: Quests are tiered (Apprentice → Journeyman → Master).
- **Idiomatic focus**: Claude evaluates not just correctness but idiomatic ObjectScript style.
- **Project-driven**: Quests build toward a cohesive application, not isolated exercises.

### CORS Configuration

IRIS must allow cross-origin requests from the Angular dev server. Add to the /api/ web application in IRIS Management Portal:

- **Allowed Origins**: http://localhost:4200 (Angular CLI default) or * for development
- **Allowed Methods**: GET, POST, OPTIONS
- **Allowed Headers**: Content-Type, Authorization

This is configured in **System Administration → Security → Web Applications → /api/** or via ObjectScript:

```objectscript
// Programmatic CORS setup (can also be done via portal)
Set props("DispatchClass") = "QuestMaster.REST.Main"
Set props("NameSpace") = "USER"
Set props("Enabled") = 1
Set props("CORSAllowed") = 1
Do ##class(Security.Applications).Modify("/api/quest/", .props)
```

### Built-in Atelier API

IRIS ships with /api/atelier/ which supports:
- PUT /api/atelier/v1/{namespace}/doc/{docname} — Create/update a document (class, routine, etc.)
- POST /api/atelier/v1/{namespace}/action/compile — Compile documents
- GET /api/atelier/v1/{namespace}/doc/{docname} — Retrieve a document

Authentication: Basic auth with default _SYSTEM / SYS (dev only).

### Custom Execution Endpoint (Quest Zero)

The Atelier API can compile but not execute arbitrary code and capture output. The player's very first quest is to create this endpoint:

```objectscript
/// QuestMaster.REST.Execute
/// REST dispatch class for compiling and executing ObjectScript code
Class QuestMaster.REST.Execute Extends %CSP.REST
{

Parameter CONTENTTYPE = "application/json";
Parameter HandleCorsRequest = 1;

XData UrlMap [ XMLNamespace = "http://www.intersystems.com/urlmap" ]
{
<Routes>
  <Route Url="/execute" Method="POST" Call="Execute"/>
  <Route Url="/compile" Method="POST" Call="Compile"/>
  <Route Url="/health" Method="GET" Call="Health"/>
</Routes>
}

ClassMethod Health() As %Status
{
  Set result = {"status": "ok", "namespace": ($Namespace)}
  Write result.%ToJSON()
  Return $$$OK
}

ClassMethod Execute() As %Status
{
  Try {
    Set body = {}.%FromJSON(%request.Content)
    Set code = body.%Get("code")
    
    // Redirect output to capture it
    Set oldIO = $IO
    Set tempFile = ##class(%File).TempFilename("txt")
    Open tempFile:"WN"
    Use tempFile
    
    Try {
      // Execute the code
      XECUTE code
    } Catch ex {
      Write "ERROR: "_ex.DisplayString()
    }
    
    // Read captured output
    Use oldIO
    Close tempFile
    
    Set output = ""
    Set file = ##class(%File).%New(tempFile)
    Do file.Open("R")
    While 'file.AtEnd {
      Set output = output _ file.ReadLine() _ $Char(10)
    }
    Do file.Close()
    Do ##class(%File).Delete(tempFile)
    
    Set result = {}
    Do result.%Set("success", 1)
    Do result.%Set("output", output)
    Write result.%ToJSON()
  } Catch ex {
    Set err = {}
    Do err.%Set("success", 0)
    Do err.%Set("error", ex.DisplayString())
    Write err.%ToJSON()
  }
  Return $$$OK
}

ClassMethod Compile() As %Status
{
  Try {
    Set body = {}.%FromJSON(%request.Content)
    Set className = body.%Get("className")
    Set source = body.%Get("source")
    
    // Save the class definition
    Set clsDef = ##class(%Dictionary.ClassDefinition).%OpenId(className)
    // ... compilation logic using %SYSTEM.OBJ.Compile()
    
    Set result = {}
    Do result.%Set("success", 1)
    Write result.%ToJSON()
  } Catch ex {
    Set err = {}
    Do err.%Set("success", 0)
    Do err.%Set("error", ex.DisplayString())
    Write err.%ToJSON()
  }
  Return $$$OK
}

}
```

**Note**: The above is a *reference implementation*. The actual Quest Zero asks the player to write this themselves — the app should validate that the /api/quest/execute endpoint is reachable before unlocking further quests.

## 4. Quest System Design

### 4.1 Difficulty Tiers

| Tier | XP Range | Focus Areas |
|------|----------|-------------|
| **Apprentice** (Levels 1–5) | 0–500 XP | Basics: WRITE, SET, IF/ELSE, FOR, QUIT, $PIECE, $LENGTH, simple globals |
| **Journeyman** (Levels 6–12) | 500–2000 XP | Classes, methods, properties, %Persistent, SQL queries, $ORDER, indirection |
| **Master** (Levels 13–20) | 2000–5000 XP | REST services, triggers, callbacks, %JSON, process management, CSP, advanced globals patterns |

### 4.2 Skill Branches

Quests are grouped into skill branches. Each branch has 5–8 quests. Completing a branch unlocks related branches.

```
                    ┌──────────────┐
                    │  Quest Zero  │
                    │ (REST Setup) │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ Commands │ │ Globals  │ │ Strings  │
        │ & Flow   │ │ & Data   │ │ & Funcs  │
        └────┬─────┘ └────┬─────┘ └────┬─────┘
             │            │            │
        ┌────▼─────┐ ┌────▼─────┐ ┌────▼─────┐
        │  Classes │ │ SQL &    │ │  %JSON & │
        │  & OOP   │ │ Queries  │ │  REST    │
        └────┬─────┘ └────┬─────┘ └────┬─────┘
             │            │            │
             └────────────┼────────────┘
                          ▼
                   ┌──────────────┐
                   │  Capstone    │
                   │  Project     │
                   └──────────────┘
```

### 4.3 Quest Structure

Each quest is a JSON object with this shape:

```typescript
interface Quest {
  id: string;                    // e.g. "globals-03"
  title: string;                 // e.g. "The Cartographer's Ledger"
  branch: string;                // e.g. "globals"
  tier: 'apprentice' | 'journeyman' | 'master';
  xpReward: number;              // e.g. 50
  bonusXP: number;               // extra XP for bonus objectives
  narrative: string;             // flavor text / story context
  objective: string;             // what the player must accomplish
  hints: string[];               // progressive hints (revealed on request)
  bonusObjectives: string[];     // optional challenges for extra XP
  expectedOutput?: string;       // if deterministic, exact expected stdout
  evaluationCriteria: string;    // instructions for Claude to evaluate the submission
  prerequisites: string[];       // quest IDs that must be completed first
  starterCode?: string;          // optional code scaffold
  conceptsIntroduced: string[];  // e.g. ["$ORDER", "global subscripts"]
}
```

### 4.4 Quest Generation via Claude API

Quests are generated dynamically by calling the Claude API. The app maintains a **quest log** (completed quests + concepts covered) and sends it as context.

**System prompt for quest generation:**

```
You are the Quest Master for an ObjectScript learning game. Generate quests that teach
InterSystems ObjectScript (COS) concepts progressively.

The player has completed these quests: {completedQuests}
They have covered these concepts: {coveredConcepts}
Their current tier is: {currentTier}
Their current skill branch is: {currentBranch}

Generate the NEXT quest in this branch. It should:
1. Introduce 1-2 new concepts while reinforcing previously learned ones
2. Have a clear, testable objective
3. Include narrative flavor text (fantasy/guild theme)
4. Include 2-3 progressive hints
5. Include 1-2 bonus objectives for extra XP
6. Specify evaluation criteria for code review

The quest's code MUST be executable on a real IRIS instance.
Use ONLY valid ObjectScript syntax.

Respond with a JSON object matching the Quest interface. No markdown fences.
```

### 4.5 Code Evaluation via Claude API

After the player submits code and it executes successfully on IRIS, the output and code are sent to Claude for evaluation.

**System prompt for evaluation:**

```
You are evaluating an ObjectScript code submission for a learning quest.

Quest: {questTitle}
Objective: {questObjective}
Evaluation Criteria: {evaluationCriteria}
Expected Output (if any): {expectedOutput}
Bonus Objectives: {bonusObjectives}

Player's code:
{playerCode}

IRIS execution output:
{executionOutput}

IRIS errors (if any):
{errors}

Evaluate the submission. Respond with JSON:
{
  "passed": boolean,
  "score": number (0-100),
  "bonusAchieved": string[] (which bonus objectives were met),
  "feedback": string (encouraging, specific feedback),
  "codeReview": string (idiomatic ObjectScript suggestions),
  "xpEarned": number
}
```

## 7. IRIS REST Communication Layer

### 7.1 IrisApiService

Create an Angular injectable service encapsulating all IRIS communication:

```typescript
// src/app/services/iris-api.service.ts
@Injectable({ providedIn: 'root' })
export class IrisApiService {
  private http = inject(HttpClient);

  // Execute arbitrary ObjectScript code via the Quest endpoint
  executeCode(config: IRISConfig, code: string): Observable<ExecuteResult>

  // Compile a class via the Atelier API
  compileClass(config: IRISConfig, className: string, source: string): Observable<ExecuteResult>

  // Health check — verifies the Quest REST endpoint is reachable
  healthCheck(config: IRISConfig): Observable<boolean>

  // Save a document (class, routine) via Atelier API
  saveDocument(config: IRISConfig, docName: string, content: string): Observable<ExecuteResult>
}
```

Supporting interfaces:

```typescript
// src/app/models/iris.models.ts
export interface IRISConfig {
  baseUrl: string;       // e.g. "http://localhost:52773"
  namespace: string;     // e.g. "USER"
  username: string;      // e.g. "_SYSTEM"
  password: string;      // e.g. "SYS"
}

export interface ExecuteResult {
  success: boolean;
  output?: string;
  error?: string;
}
```

### 7.2 Authentication

All IRIS REST calls use HTTP Basic Auth. Implement via an HttpInterceptor or directly in the service:

```typescript
private getHeaders(config: IRISConfig): HttpHeaders {
  return new HttpHeaders({
    'Content-Type': 'application/json',
    'Authorization': 'Basic ' + btoa(`${config.username}:${config.password}`)
  });
}
```

### 7.3 Connection Status Indicator

The UI should show a persistent connection indicator:
- 🟢 Connected (health check passes)
- 🔴 Disconnected (health check fails)
- 🟡 Connecting (health check in progress)

Use an Angular signal for status and poll with interval() from RxJS every 30 seconds:

```typescript
// src/app/services/iris-connection.service.ts
@Injectable({ providedIn: 'root' })
export class IrisConnectionService {
  connectionStatus = signal<'connected' | 'disconnected' | 'connecting'>('disconnected');

  startPolling(config: IRISConfig): void {
    interval(30000).pipe(
      startWith(0),
      switchMap(() => this.irisApi.healthCheck(config))
    ).subscribe(/* update signal */);
  }
}
```

## 8. Claude API Integration

### 8.1 ClaudeApiService

```typescript
// src/app/services/claude-api.service.ts
@Injectable({ providedIn: 'root' })
export class ClaudeApiService {

  async callClaude(systemPrompt: string, userMessage: string, apiKey: string): Promise<any> {
    const response = await fetch('https://api.anthropic.com/v1/messages', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'x-api-key': apiKey,
        'anthropic-version': '2023-06-01',
        'anthropic-dangerous-direct-browser-access': 'true'
      },
      body: JSON.stringify({
        model: 'claude-sonnet-4-20250514',
        max_tokens: 2048,
        system: systemPrompt,
        messages: [{ role: 'user', content: userMessage }]
      })
    });
    return response.json();
  }

  generateQuest(completedQuests: string[], coveredConcepts: string[], currentBranch: string, tier: string, apiKey: string): Promise<Quest>

  evaluateSubmission(quest: Quest, playerCode: string, executionOutput: string, errors: string, apiKey: string): Promise<EvaluationResult>
}
```

**Important**: The header `anthropic-dangerous-direct-browser-access` is required for client-side calls. The user must supply their own API key. Note: we use fetch here rather than Angular HttpClient because the Anthropic API requires custom headers that Angular's XSRF protection can interfere with.

### 8.2 Two Claude Functions

1. **generateQuest(completedQuests, coveredConcepts, currentBranch, tier)** — Returns a Quest JSON object
2. **evaluateSubmission(quest, playerCode, executionOutput, errors)** — Returns evaluation JSON

### 8.3 Fallback Behavior

If the Claude API is unreachable or the user hasn't entered an API key:
- Use only the hard-coded starter quests (Quest 0, 1, 2, and any others defined in a local quest bank)
- Skip AI code review — just check if IRIS execution succeeded without errors
- Show a banner: "AI features disabled. Add your Anthropic API key in Settings to unlock quest generation and code review."

## 11. Monaco Editor — ObjectScript Syntax

Monaco doesn't ship with ObjectScript support. Register a custom language in the CodeEditorComponent's initialization:

```typescript
monaco.languages.register({ id: 'objectscript' });
monaco.languages.setMonarchTokensProvider('objectscript', {
  keywords: [
    'SET', 'WRITE', 'IF', 'ELSE', 'FOR', 'WHILE', 'DO', 'QUIT', 'RETURN',
    'NEW', 'KILL', 'HANG', 'OPEN', 'CLOSE', 'USE', 'READ', 'TRY', 'CATCH',
    'THROW', 'LOCK', 'MERGE', 'XECUTE', 'GOTO', 'JOB',
    'set', 'write', 'if', 'else', 'for', 'while', 'do', 'quit', 'return',
    'new', 'kill', 'hang', 'open', 'close', 'use', 'read', 'try', 'catch',
    'throw', 'lock', 'merge', 'xecute', 'goto', 'job',
    // Short forms
    's', 'w', 'i', 'e', 'f', 'd', 'q', 'n', 'k', 'h'
  ],
  intrinsicFunctions: [
    '$PIECE', '$LENGTH', '$EXTRACT', '$FIND', '$ORDER', '$DATA', '$GET',
    '$INCREMENT', '$LIST', '$LISTBUILD', '$LISTLENGTH', '$LISTGET',
    '$ZSTRIP', '$TRANSLATE', '$REPLACE', '$REVERSE', '$JUSTIFY',
    '$HOROLOG', '$CLASSMETHOD', '$CLASSNAME', '$SYSTEM',
    '$piece', '$length', '$extract', '$find', '$order', '$data', '$get',
    '$p', '$l', '$e', '$f', '$o', '$d', '$g', '$i',
    '$lb', '$li', '$ll'
  ],
  tokenizer: {
    root: [
      [/\/\/.*$/, 'comment'],
      [/;.*$/, 'comment'],
      [/#;.*$/, 'comment'],
      [/\^[%A-Za-z][A-Za-z0-9.]*/, 'variable.global'],
      [/\$\$[%A-Za-z][A-Za-z0-9]*/, 'function.user'],
      [/\$[A-Za-z]+/, 'function.builtin'],
      [/"[^"]*"/, 'string'],
      [/\b\d+\.?\d*\b/, 'number'],
      [/[A-Z]{2,}(?=\s|$|\()/, { cases: { '@keywords': 'keyword' }}],
      [/\b(Class|Property|Method|Parameter|ClassMethod|Relationship|Index|Trigger|Query|Storage)\b/, 'keyword.class'],
      [/##class/i, 'keyword.class'],
    ]
  }
});
```

## 12. Angular-Specific Implementation Notes

### 12.1 Standalone Components

Use Angular standalone components throughout — no NgModules. All components should declare standalone: true and import their dependencies directly.

### 12.2 Signals for Reactive State

Use Angular signals (signal(), computed(), effect()) for all state management rather than BehaviorSubjects. This aligns with Angular's modern reactivity model.

### 12.3 Application Config

```typescript
// src/app/app.config.ts
import { ApplicationConfig } from '@angular/core';
import { provideHttpClient } from '@angular/common/http';
import { provideAnimations } from '@angular/platform-browser/animations';

export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(),
    provideAnimations()
  ]
};
```

### 12.4 Angular Proxy for Development

To avoid CORS issues during development, configure the Angular CLI proxy:

```json
// proxy.conf.json
{
  "/api/quest": {
    "target": "http://localhost:52773",
    "secure": false,
    "changeOrigin": true
  },
  "/api/atelier": {
    "target": "http://localhost:52773",
    "secure": false,
    "changeOrigin": true
  }
}
```

Start with: ng serve --proxy-config proxy.conf.json

This means during development the app calls /api/quest/execute (same origin) and Angular CLI proxies it to IRIS. In production you'd configure IRIS CORS directly.

### 12.5 Quest Zero Validation

Before unlocking any quest beyond Quest Zero, the app should:
1. Call GET /api/quest/health — verify it returns 200
2. Call POST /api/quest/execute with {"code": "WRITE 1+1"} — verify output contains "2"

If either fails, show instructions for completing Quest Zero.
