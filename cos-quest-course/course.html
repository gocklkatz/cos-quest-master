A gamified ObjectScript learning platform that teaches programming through quests, coding challenges, and instant feedback

The Outside World

How the app talks to IRIS, Claude API, and what happens "behind the curtain"

Two External Services

COS Quest Master connects to two external systems:

🐳

InterSystems IRIS

The database and application server where your ObjectScript code runs. It's containerized in Docker for easy setup.

# curl command from the browser
POST /api/quest/execute
Body: { "code": "WRITE 1+1" }
# Response:
{ "success": true, "output": "2\n" }
🧠

Claude API

Generates quests and evaluates your code. Your API key is stored locally and sent directly to Anthropic.

# API call structure
POST https://api.anthropic.com/v1/messages
Authorization: Bearer sk-ant-...
Content-Type: application/json
Body: { "model": "claude-sonnet-4-6", ... }

IRIS REST Endpoints

IRIS exposes a REST API with three main endpoints:

🏥

GET /api/quest/health

Checks if IRIS is reachable. Returns {"status":"ok","namespace":"USER"}. Used to show connection status in the header.

⚡

POST /api/quest/execute

Executes ObjectScript code. Sends code as JSON, returns success boolean and captured output. This is your code's main execution path.

💾

POST /api/quest/compile

Compiles ObjectScript classes (.cls files). Used for quests that involve defining new classes or methods.

CORS and the Proxy Conundrum

Browser security prevents direct API calls from localhost:4200 to localhost:52773. Here's how the app handles it:

// angular.json devServer config
{
  "proxyConfig": "proxy.conf.json"
}

// proxy.conf.json
{
  "/api/quest": {
    "target": "http://localhost:52773",
    "secure": false,
    "changeOrigin": true
  }
}

The dev server proxies requests to IRIS

Angular sends /api/quest/execute (same origin)

CLI forwards it to the IRIS Docker container

The app appears to make same-origin requests

ℹ️
Production vs Development

In production, you'd configure CORS headers on IRIS instead of using a proxy. The proxy only exists in the Angular dev server.

What's in the IRIS REST Class?

The ObjectScript REST class handles requests. Here's the core structure:

Class QuestMaster.REST.Execute Extends %CSP.REST
{
Parameter CONTENTTYPE = "application/json";
Parameter HandleCorsRequest = 1;

// Route definitions
XData UrlMap
{
<Routes>
  <Route Url="/execute" Method="POST" Call="Execute"></Route>
  <Route Url="/health" Method="GET" Call="Health"></Route>
  <Route Url="/compile" Method="POST" Call="Compile"></Route>
</Routes>
}

Method Execute() As %Status
{
  // Get request body
  Set body = {}.%FromJSON(%request.Content)
  Set code = body.%Get("code")

  // Redirect IO to capture output
  Set oldIO = $IO
  Set tempFile = ##class(%File).TempFilename("txt")
  Open tempFile:"WN"
  Use tempFile

  Try {
    XECUTE code  // Execute the ObjectScript
  } Catch ex {
    Write "ERROR: " _ ex.DisplayString()
  }

  // Restore and read captured output
  Use oldIO
  Close tempFile
  // ... read file, return JSON response
}

Extends %CSP.REST to become a REST endpoint

UrlMap XData defines available routes

XECUTE runs the ObjectScript code

$IO redirection captures WRITE output

CATCH handles errors gracefully

The Clever Tricks

Engineering patterns and clever solutions that make the app work smoothly

Signals: Reactive State

Angular Signals are the heart of COS Quest Master. They make state management simple yet powerful.

💡
Why Signals Beat Old-School State

Before signals, you'd use observables or manually call change detection. Signals automatically update dependent values. No more forgotten "detectChanges()" calls!

Before Signals (Old Pattern):

export class GameStateService {
  private xpSubject = new BehaviorSubject(0);
  xp$ = this.xpSubject.asObservable();

  updateXp(amount: number) {
    const newXp = this.xpSubject.getValue() + amount;
    this.xpSubject.next(newXp);
  }
}

Create a BehaviorSubject to hold state

Convert to Observable for components

Manually get current value, update, and push

Components subscribe to get updates

After Signals (Modern Pattern):

export class GameStateService {
  private state = signal<GameState>(new GameState());

  // Selectors
  xp = computed(() => this.state().xp);
  level = computed(() => calcLevel(this.xp()));

  // Mutations
  completeQuest(questId: string) {
    this.state.update(s => ({
      ...s,
      xp: s.xp + quest.xpReward,
      completedQuests: [...s.completedQuests, questId]
    }));
  }
}

Signal holds the state object

Computed signals auto-calculate from state

Update applies changes immutably

Any reading signal updates automatically

Key benefits:

  • No subscription cleanup needed
  • No manual change detection
  • Computed values are cached automatically
  • Update is atomic and safe

localStorage: Browser File System

Your progress is saved to localStorage — a small database that lives in your browser. Even after closing the tab, your XP, completed quests, and settings remain.

// The storage key
const STORAGE_KEY = 'questmaster';

// Load from storage on startup
private loadFromStorage(): GameState {
  try {
    const raw = localStorage.getItem(STORAGE_KEY);
    if (raw) {
      return { ...JSON.parse(raw) };
    }
  } catch {
    // Ignore errors
  }
  return new GameState();
}

// Save to storage after every change
private persist() {
  localStorage.setItem(
    STORAGE_KEY,
    JSON.stringify(this.state())
  );
}

localStorage is key-value storage in browser

Data persists across page reloads

All state is serialized to JSON

Changes are saved immediately

File Tabs & Execution Model

COS Quest Master handles both class files (.cls) and script files (.script) in a unified way.

// Quest files array
const files = [
  {
    id: 'cls-main',
    filename: 'User.GuildMember.cls',
    fileType: 'cls',
    label: 'Class Definition',
    starterCode: 'Class User.GuildMember Extends %Persistent...'
  },
  {
    id: 'main',
    filename: 'solution.script',
    fileType: 'script',
    label: 'Solution',
    dependsOn: ['cls-main'],  // Compile class first
    starterCode: 'SET member = ##class(User.GuildMember).%New()...'
  }
];

// Execution order:
// 1. Compile all .cls files in order
// 2. Execute all .script files in order

Each quest defines multiple files

Files have unique IDs for tab navigation

dependsOn ensures proper compilation order

Class files compile first, scripts second

When Things Break

Debugging common issues, understanding errors, and escaping AI bug loops

Three Categories of Errors

When your code doesn't work, the error falls into one of these categories. Knowing which helps you debug faster.

Syntax Error

The code is malformed. IRIS can't even parse it.

WRITE "Hello World  // Missing closing quote

Runtime Error

Code parses but fails when executed.

SET x = 10 / 0  //

Logic Error

Code runs but produces wrong output. Hardest to detect!

FOR i = 1:1:count  // Off-by-one error

Debugging Flow: Your First 5 Minutes

When something breaks, follow this path:

1
Read the error message

IRIS errors look like <ERROR>: <DIVIDE>. The error type tells you the category.

2
Locate the line number

Errors include line numbers. Check the code editor gutter or scroll to find it.

3
Check for common issues

Missing quotes, unmatched brackets, wrong function names, division by zero.

4
Use WRITE for debugging

Add WRITE "Debug: x = ", x, ! to trace values before the error.

5
Break it down

Simplify the code. Test smaller pieces until the error disappears.

Common ObjectScript Bugs

These bugs trip up beginners constantly. Recognizing them helps you fix issues faster:

// BUG 1: Missing quotes in WRITE
WRITE Hello World, !  // ERROR: 'Hello' undefined variable

// FIX: Wrap string in quotes
WRITE "Hello World", !

ObjectScript requires quoted strings

Unquoted text is treated as variable names

// BUG 2: Wrong string concatenation
WRITE "Hello" "World", !  // Outputs: "Hello" "World"

// FIX: Use underscore (_) for concatenation
WRITE "Hello " _ "World", !  // Outputs: Hello World

Adjacent strings aren't concatenated automatically

Use _ operator to join strings

// BUG 3: Missing % in method calls
SET obj = MyClass.New()  // ERROR: 'New' undefined function

// FIX: Use ##class() and % prefix for methods
SET obj = ##class(MyClass).%New()

Instance methods need .%MethodName() syntax

Class methods use ##class().MethodName()

Escaping the AI Bug Loop

When AI keeps generating broken code, try these strategies:

🔍

Read the Error First

Before asking AI to fix it, understand what the error means. The error line number is your anchor.

🧩

Simplify the Code

Create a minimal example that reproduces the issue. AI struggles with complex, multi-file bugs.

📝

Ask for Explanations

Instead of "fix this," ask "why does this error occur?" Understanding beats copy-paste.

🧪

Test Small Pieces

Compile and run individual classes before testing the full integration. Isolate the problem.
